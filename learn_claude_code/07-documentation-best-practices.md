# 7. ドキュメントのベストプラクティス

この章は「売れているドキュメント（読みやすく、導入しやすいドキュメント）」の
構成パターンを抽出したものです。

## 7.1 Diataxis の 4 分類
Diataxis ではドキュメントを 4 種類に分けることが推奨されています。

- **Tutorial**: 初学者向けの導入体験
- **How-to**: 具体的な目的を達成する手順
- **Reference**: 仕様や API の正確な記述
- **Explanation**: 背景や仕組みの理解

**ポイント**: 4 種類を混ぜない。
「チュートリアルに仕様を詰め込む」ことが最大の失敗パターンです。

## 7.2 売れているドキュメントの構成パターン
多くの成功しているプロダクトは、以下の順序で構成されています。

1. **Overview / What is this?**
2. **Quickstart**
3. **Core Concepts**
4. **How-to Guides**
5. **Reference**
6. **FAQ / Troubleshooting**

この順序は「理解 → 体験 → 深掘り → 詳細」の流れになっています。

## 7.2.1 良い / 悪い例（構成）

| 観点 | 悪い例 | 良い例 |
| --- | --- | --- |
| 導入 | いきなり詳細仕様 | Quickstart で最短の成功体験 |
| 構成 | Tutorial と Reference が混ざる | Tutorial / Reference を分離 |
| 入口 | 何をすればいいか不明 | 最初の 5 行で目的がわかる |

## 7.3 書き方の原則
- **明確さ**: 1 文 1 目的、曖昧語を避ける
- **一貫性**: 用語、表記、命名を揃える
- **アクセシビリティ**: 読みやすい構造、視認性
- **利用者目線**: 読み手の目的から逆算する

## 7.4 情報設計（IA）の基本
- **入口を用意する**: Quickstart は最短で成功体験へ
- **階層を浅く**: 迷わないナビゲーション
- **検索性を高める**: Reference を分離

## 7.5 Good Docs Project のテンプレ設計
Good Docs Project では、以下のテンプレが用意されています。

- Overview
- Tutorial
- How-to
- Reference
- Troubleshooting

これらは「何を書くべきか」を明確にするための枠組みです。

## 7.6 文章の品質を上げるチェック
- 最初の 5 行で「何ができるか」が伝わるか
- 用語は最初に定義されているか
- 例やコードが 1 つ以上あるか
- 読み手が次に何をすべきかが明確か

## 7.6.1 演習テンプレ（ドキュメント断片）
目的: 1 画面で完結するドキュメント断片を書く
入力: 実務の小さな作業 1 件
出力:
```
# タイトル

## 目的

## 手順
1. 
2. 

## 期待結果

## 参考
```

### 解答例
```
# エラーメッセージ統一

## 目的
ログイン画面のエラーメッセージを 1 つに統一する。

## 手順
1. `app/ui/login/` の文言定義を確認する
2. 文言を共通関数に集約する

## 期待結果
同じ条件で同一メッセージが表示される。

## 参考
docs/ui-guidelines.md
```

## 7.7 ドキュメントの運用
- 更新ルールを決める（誰が、いつ、どう更新するか）
- バージョンを明記する
- 古い情報を削除する

## 7.8 Claude Code ドキュメントに活かす
- Overview / Quickstart / Concepts / Features / Practical / MCP 連携の順に並べる
- Tutorial と Reference を分離する
- チェックリストと練習問題を入れて定着を促す
